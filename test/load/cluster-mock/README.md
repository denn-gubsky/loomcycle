# cluster-mock — multi-replica mock-mode stress harness

Cluster analogue of `circuit-stress/` (which exercises ONE binary against
mock-mode). Brings up an N-replica loomcycle cluster (built from HEAD,
mock provider, nginx LB, co-resident Postgres), drives the same
`circuit-stress` harness through the LB, and reads the shared cluster
Postgres for per-replica rollups.

Single-replica characterisation lives in
`~/work/loomcycle-internal/doc-internal/research/load-test-2026-05-27/`
(Findings §1-§7). This rig produces the multi-replica data for §8.

## What it tests

The same 3-agent pipeline (`researcher → editor → evaluator`) the
single-replica harness drives, with each agent spawned via a separate
POST through the LB. Because nginx round-robins, the three agents of a
single circuit usually land on different replicas — which forces every
piece of cross-replica coordination through real traffic:

| Substrate | What gets exercised |
|---|---|
| `runs.replica_id` | Ownership stamping at create; needed for cancel routing + crash reap |
| `loomcycle.channel` LISTEN/NOTIFY | Editor on replica B subscribes to a channel the researcher published on replica A → fan-out must traverse the backplane |
| `cancel_coordinator` + `loomcycle.cancel`/`.ack` | Cancel posted to replica B for a run owned by replica A → broadcast + ack round-trip |
| `replicas` table + dead-replica reaper | A killed replica's in-flight runs marked `failed`/`owner_replica_dead`; quotas reclaimed |
| `process_samples.replica_id` (Phase 1 of this branch) | Per-replica CPU/RSS series in the shared table — see `feat(metrics): tag process_samples with replica_id` |
| pgxpool budget split | Per-replica pool sized from N so `N × pool ≤ max_connections` — surfaces the central-Postgres connection budget as a real constraint |

## Quickstart

```sh
export LOOMCYCLE_AUTH_TOKEN=loomcycle   # bearer + Web UI token

# Smoke — 2 replicas × 100 circuits, no failure injection.
./test/load/cluster-mock/run-cluster-mock.sh --replicas 2 --scale 100

# Throughput @ 15% 429 — the headline scaling test.
LOOMCYCLE_MOCK_429_RATE=0.15 \
  ./test/load/cluster-mock/run-cluster-mock.sh --replicas 3 --scale 5000 \
  --circuits-per-user 20 --circuit-timeout 10m

# Cross-replica cancel under load (injector runs concurrently).
LOOMCYCLE_MOCK_429_RATE=0.15 \
  ./test/load/cluster-mock/run-cluster-mock.sh --replicas 3 --scale 5000 \
  --scenario cancel

# Replica-crash recovery — docker-kills replica-2 mid-load, verifies
# the dead-replica reaper fires within the run window. The run script
# auto-tunes LOOMCYCLE_REPLICAS_SWEEP_INTERVAL_MS=5000 and
# LOOMCYCLE_REPLICAS_STALE_AFTER_MS=15000 for the crash scenario only.
LOOMCYCLE_MOCK_429_RATE=0.15 \
  ./test/load/cluster-mock/run-cluster-mock.sh --replicas 3 --scale 5000 \
  --scenario crash

# Web UI of the LB during a run (--no-teardown to keep the cluster up):
#   open "http://127.0.0.1:18080/ui?token=$LOOMCYCLE_AUTH_TOKEN"
```

## Topology (generated by `run-cluster-mock.sh`)

```
        host :15433  ◀──────────────  postgres:5432   (cluster state + process_samples)
                                          ▲
                                          │ shared pgxpool (per-replica = floor(160/N))
                            ┌─────────────┼─────────────┐
                            │             │             │
                       replica-1     replica-2     ... replica-N         (mock provider, LOOMCYCLE_REPLICA_ID set)
                    host :18801  host :18802  ... host :18800+N
                            │             │             │
                            └─────────────┴─────────────┘
                                          ▲
                                          │ round-robin (no sticky sessions)
                                      nginx LB
                                  host :18080  ◀── circuit-stress driver
```

Per-replica pgxpool is computed by the run script from N so total
connections stay under `max_connections=200` (`floor(160/N)`, min 10).
Cluster-wide capacity = `N × per-replica pool` — splitting the central
Postgres budget is the binding admission constraint, not the per-replica
`max_concurrent_runs`.

## Flags & env

| Flag | Default | Notes |
|---|---|---|
| `--replicas N` | `3` | Number of loomcycle replicas in the generated compose |
| `--scale M` | `1` | Circuits the driver launches (passed through) |
| `--scenario {none,cancel,crash}` | `none` | Concurrent injector to run alongside the driver |
| `--no-teardown` | off | Leave the cluster up after the run for inspection |
| (any other) | — | Passed through to the `circuit-stress` driver |

Failure-injection + sampler env vars (same as `circuit-stress`):
`LOOMCYCLE_MOCK_429_RATE`, `LOOMCYCLE_MOCK_500_RATE`,
`LOOMCYCLE_MOCK_LATENCY_MS`, `LOOMCYCLE_MOCK_LATENCY_JITTER_MS`,
`LOOMCYCLE_CHANNEL_DEBUG=1`.

Replicas-sweeper tuning (defaults match production):
`LOOMCYCLE_REPLICAS_SWEEP_INTERVAL_MS`, `LOOMCYCLE_REPLICAS_STALE_AFTER_MS`
— wired by [feat(coord): tunable replicas-sweeper interval + stale-after]
in this branch. The `--scenario crash` path auto-sets `5000 / 15000` so
the reap fires inside a sub-minute load run.

## Per-replica analysis

`analyze-cluster.sh` reads the cluster Postgres directly (the same way
the prior single-replica analysis read `process_samples`). The full
output is written to `cluster-analysis.txt`; the rollups it computes:

| Rollup | What it proves |
|---|---|
| `runs GROUP BY replica_id, status` | LB round-robin + ownership stamping work; load splits cleanly |
| `process_samples GROUP BY replica_id` | The Phase-1 `replica_id` column populates per replica → per-replica CPU/RSS/goroutines attributable |
| Cross-replica circuit count | Circuits whose 3 agents span ≥2 replicas → editor/evaluator subscribe traversed `loomcycle.channel` NOTIFY |
| `stop_reason IN ('owner_replica_dead','heartbeat_stale')` | Dead-replica reaper fired (crash scenario) |
| `count(*) WHERE status='running'` | Zombie-run guard: should be 0 after every test |
| `sum(user_quotas.active_count)` | Quota-leak guard: should be 0 after a clean drain |

Standalone use (against a `--no-teardown` cluster):

```sh
PG_DSN="postgres://postgres:loomcycle@127.0.0.1:15433/loomcycle?sslmode=disable" \
  RESULTS_DIR=/tmp/x REPLICAS=3 \
  bash test/load/cluster-mock/analyze-cluster.sh
```

## Known limitations

- **Single Postgres host.** This rig shares one DB across all replicas
  (it's the v0.12.x cluster shape per `docs/MULTI-REPLICA.md`).
  Multi-region with read replicas is post-v1.0 scope and not covered here.
- **`max_concurrent_runs_per_user: 0`** (disabled) — fairness behavior
  isn't on the test path. Set explicitly in the yaml to exercise it.
- **Image-build cost.** First run rebuilds `loomcycle:mock-local` from HEAD
  (~1-2 min for the Node + Go stages). Subsequent runs use the layer cache.
- **No replica scale-down test.** Adding `--replicas N+1` mid-run isn't
  supported by the generator — replica count is fixed per compose.
